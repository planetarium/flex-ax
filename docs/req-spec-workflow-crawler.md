# flex 워크플로우 데이터 크롤러 요구 명세서

## 1. 개요

### 1.1 배경 및 목적

flex(HR SaaS)는 전자결재 기반의 워크플로우 기능을 제공하며, 근태 승인, 휴가 신청, 경비 승인 등 다양한 업무 프로세스를 처리한다. 현재 flex 내에 축적된 워크플로우 데이터를 외부에서 체계적으로 수집하여 별도 저장 및 활용할 수 있는 수단이 필요하다.

본 요구사항은 flex 서비스의 워크플로우 데이터를 수집(크롤링)하여 로컬에 저장하는 일회성 스크립트에 대한 것이다.

### 1.2 해결하려는 문제

- flex 내 워크플로우 데이터를 flex 서비스 외부에서 활용할 수 없는 상황
- 워크플로우의 정의(템플릿)와 실행 이력(인스턴스)을 통합적으로 파악할 수 있는 수단 부재

## 2. 범위

### 2.1 포함 범위 (In-Scope)

- flex에서 제공하는 **모든 종류의 워크플로우** 데이터 수집
  - 전자결재 워크플로우 (품의, 구매요청, 지출결의서, 업무협조전 등 커스텀 양식 포함)
  - 근태 관련 승인 (근무 변경, 초과근무 신청 등)
  - 휴가 관련 승인 (연차, 법정휴가, 맞춤휴가 등)
  - 경비/지출 관련 승인
  - 기타 flex에서 승인 프로세스를 거치는 모든 항목
- 워크플로우 **정의/템플릿** 데이터 수집 (양식 구성, 필드 정의, 결재선 설정 등)
- 워크플로우 **실행 이력/인스턴스** 데이터 수집 (신청 내역, 승인/반려 상태, 처리 이력 등)
- 수집된 데이터의 로컬 저장
- 단일 테넌트(회사) 대상

### 2.2 제외 범위 (Out-of-Scope)

- 수집된 데이터의 분석, 가공, 시각화
- 주기적/자동화된 반복 실행 (스케줄링)
- 다중 테넌트 지원
- flex의 워크플로우 외 다른 기능(급여, 채용 등) 데이터 수집
- 수집된 데이터를 flex에 다시 쓰는 행위 (읽기 전용)

## 3. 용어 정의

| 용어 | 정의 |
|------|------|
| **flex** | 플렉스팀(flex.team)이 제공하는 HR SaaS 서비스 |
| **워크플로우** | flex에서 제공하는 전자결재 및 승인 프로세스 기능의 총칭. 넓은 의미로 근태/휴가 승인 등도 포함 |
| **워크플로우 양식(템플릿)** | 관리자가 정의한 문서 양식. 필드 구성, 결재선 기본값, 분류 등의 정보를 포함 |
| **워크플로우 인스턴스** | 구성원이 양식을 기반으로 실제 작성/제출한 개별 결재 문서 |
| **결재선** | 문서의 승인 흐름을 정의하는 단계별 승인자/참조자 목록 |
| **결재 상태** | 워크플로우 인스턴스의 진행 상황 (예: 진행중, 승인 완료, 반려 등) |
| **테넌트** | flex 서비스에서 하나의 회사(조직) 단위를 의미하는 격리된 데이터 영역 |
| **크롤링** | 본 문서에서는 API 호출, 웹 스크래핑 등 가용한 모든 수단을 통해 데이터를 수집하는 행위를 통칭 |
| **필드** | 워크플로우 양식 내 개별 입력 항목 (텍스트, 숫자, 금액, 날짜, 선택, 복수 선택, 첨부파일 등) |

## 4. 사용자 및 역할

| 역할 | 설명 | 권한 범위 |
|------|------|-----------|
| **스크립트 실행자** | 본 스크립트를 실행하는 사람. flex에 로그인 가능한 계정을 보유 | 해당 계정이 flex에서 접근 가능한 범위의 데이터만 수집 가능 |

- 스크립트 실행자는 1명이며, 본인의 flex 계정 권한으로 접근 가능한 데이터만 수집 대상이 된다.
- 실행자의 flex 내 역할(최고관리자, 관리자, 일반 구성원 등)에 따라 수집 가능한 데이터 범위가 달라질 수 있다.

## 5. 기능 요구사항

### 5.1 데이터 접근 경로 탐색

- **FR-001**: 시스템은 flex 서비스에서 워크플로우 데이터에 접근할 수 있는 모든 가용 경로를 활용할 수 있어야 한다.
  - 선행 조건: 없음 (설계 시 고려사항)
  - 기대 결과: 다음 우선순위에 따라 데이터 접근 경로를 결정한다.
    1. **flex Open API**: flex가 공식 제공하는 Open API에 워크플로우 관련 endpoint가 존재하는 경우 이를 우선 사용
    2. **웹 UI 크롤링**: Open API로 충족되지 않는 데이터는 flex 웹 애플리케이션의 UI를 통해 수집
    3. **내부 API 활용**: flex 웹 애플리케이션이 내부적으로 호출하는 API가 식별되는 경우 이를 활용
  - 예외 상황: 어떤 경로로도 접근할 수 없는 데이터가 존재할 경우, 해당 데이터 항목과 접근 불가 사유를 로그로 기록한다.
  - 비고: flex Open API는 현재 인사정보, 조직도, 근무/휴가 이력 중심으로 제공되며, 워크플로우 전용 API의 존재 여부는 확인이 필요하다 (미결 사항 참조).

### 5.2 인증

- **FR-002**: 시스템은 flex 서비스에 인증하여 데이터에 접근할 수 있어야 한다.
  - 선행 조건: 실행자가 유효한 flex 계정 정보를 보유
  - 기대 결과: flex 서비스에 성공적으로 인증하고, 인증 상태를 유지하며 데이터를 수집할 수 있다.
  - 예외 상황:
    - 인증 실패 시 명확한 오류 메시지를 출력하고 스크립트를 종료한다.
    - 세션 만료 시 재인증을 시도한다.

- **FR-003**: 인증에 필요한 민감 정보(계정 정보, 토큰 등)는 스크립트 코드에 직접 포함되지 않아야 한다.
  - 선행 조건: 없음
  - 기대 결과: 환경 변수, 설정 파일 등 코드 외부에서 인증 정보를 주입받는다.

### 5.3 워크플로우 정의/템플릿 수집

- **FR-004**: 시스템은 대상 테넌트에 존재하는 모든 워크플로우 양식(템플릿) 목록을 수집할 수 있어야 한다.
  - 선행 조건: 인증 완료 (FR-002)
  - 기대 결과: 실행자의 권한으로 조회 가능한 모든 양식 목록이 수집된다.
  - 예외 상황: 권한 부족으로 조회되지 않는 양식이 있을 수 있으며, 이는 허용되는 동작이다.

- **FR-005**: 각 워크플로우 양식에 대해 다음 정보를 수집해야 한다 (해당 데이터가 접근 가능한 경우).
  - 양식 식별자 (고유 ID 또는 이름)
  - 양식 분류/카테고리
  - 양식에 포함된 필드 목록과 각 필드의 속성 (필드명, 필드 유형, 필수 여부 등)
  - 기본 결재선 설정 (승인 단계, 각 단계의 승인 유형 등)
  - 양식 생성/수정 일시 (가용한 경우)
  - 양식의 사용 권한 설정 (가용한 경우)
  - 선행 조건: FR-004 완료
  - 기대 결과: 각 양식의 상세 정보가 구조화된 형태로 수집된다.
  - 예외 상황: 특정 필드 정보가 접근 불가한 경우 해당 필드는 누락으로 표시하고 나머지 정보는 정상 수집한다.

### 5.4 워크플로우 실행 이력/인스턴스 수집

- **FR-006**: 시스템은 대상 테넌트에 존재하는 워크플로우 인스턴스(결재 문서) 목록을 수집할 수 있어야 한다.
  - 선행 조건: 인증 완료 (FR-002)
  - 기대 결과: 실행자의 권한으로 조회 가능한 워크플로우 인스턴스 목록이 수집된다.
  - 예외 상황:
    - 대량의 인스턴스가 존재할 경우 페이지네이션 등을 통해 누락 없이 수집한다.
    - 권한 부족으로 조회되지 않는 인스턴스가 있을 수 있으며, 이는 허용되는 동작이다.

- **FR-007**: 각 워크플로우 인스턴스에 대해 다음 정보를 수집해야 한다 (해당 데이터가 접근 가능한 경우).
  - 인스턴스 식별자 (고유 ID 또는 문서 번호)
  - 사용된 양식 정보 (FR-005에서 수집한 양식과의 연결)
  - 기안자 정보 (이름, 소속 등)
  - 기안 일시
  - 현재 결재 상태 (진행중, 승인 완료, 반려 등)
  - 결재선 상세 (각 단계별 승인자, 승인/반려 일시, 코멘트)
  - 문서 본문 (각 필드에 입력된 값)
  - 첨부파일 정보 (파일명, 가능한 경우 파일 자체)
  - 문서 수정 이력 (가용한 경우)
  - 선행 조건: FR-006 완료
  - 기대 결과: 각 인스턴스의 상세 정보가 구조화된 형태로 수집된다.
  - 예외 상황: 특정 항목이 접근 불가한 경우 해당 항목은 누락으로 표시하고 나머지 정보는 정상 수집한다.

### 5.5 근태/휴가 승인 데이터 수집

- **FR-008**: 시스템은 전자결재 워크플로우 외에 flex에서 별도로 관리되는 근태/휴가 승인 프로세스의 데이터도 수집할 수 있어야 한다.
  - 선행 조건: 인증 완료 (FR-002)
  - 기대 결과: 근무 변경 신청/승인, 초과근무 신청/승인, 휴가 신청/승인 등의 이력이 수집된다.
  - 예외 상황: flex의 근태/휴가 승인이 워크플로우와 동일 메커니즘으로 처리되는 경우, FR-006/FR-007에서 함께 수집될 수 있다. 이 경우 중복 수집하지 않는다.
  - 비고: flex에서 근태/휴가 승인은 워크플로우(전자결재)와 별도 메뉴에서 관리되는 것으로 파악된다. 실제 내부 데이터 구조가 동일한지는 확인이 필요하다 (미결 사항 참조).

### 5.6 데이터 저장

- **FR-009**: 수집된 모든 데이터는 로컬 파일 시스템에 저장되어야 한다.
  - 선행 조건: 데이터 수집 완료 (FR-004 ~ FR-008 중 하나 이상)
  - 기대 결과: 수집된 데이터가 사람이 읽을 수 있고, 프로그래밍적으로 처리 가능한 형태로 저장된다.
  - 예외 상황: 저장 공간 부족 시 오류 메시지를 출력하고 스크립트를 종료한다.

- **FR-010**: 저장된 데이터는 워크플로우 정의(템플릿)와 실행 이력(인스턴스)을 구분하여 저장해야 한다.
  - 선행 조건: FR-009
  - 기대 결과: 양식 정보와 인스턴스 정보가 논리적으로 분리되어 있으며, 인스턴스에서 해당 양식을 참조할 수 있는 연결 정보가 포함된다.

- **FR-011**: 첨부파일이 수집된 경우, 첨부파일은 별도의 디렉토리에 저장되고 해당 인스턴스 데이터에서 참조할 수 있어야 한다.
  - 선행 조건: FR-007에서 첨부파일 수집 성공
  - 기대 결과: 첨부파일 원본이 저장되며, 인스턴스 데이터에 파일 참조 경로가 포함된다.

### 5.7 실행 제어 및 로깅

- **FR-012**: 스크립트 실행 시 진행 상황을 확인할 수 있어야 한다.
  - 선행 조건: 스크립트 실행 중
  - 기대 결과: 현재 수집 단계, 처리 건수, 오류 발생 여부 등이 실행 중 출력된다.

- **FR-013**: 스크립트 실행 완료 후 수집 결과 요약을 확인할 수 있어야 한다.
  - 선행 조건: 스크립트 실행 완료
  - 기대 결과: 수집된 양식 수, 인스턴스 수, 실패 건수, 소요 시간 등의 요약 정보가 출력된다.

- **FR-014**: 수집 중 개별 항목에서 오류가 발생하더라도 나머지 항목의 수집은 계속되어야 한다.
  - 선행 조건: 스크립트 실행 중
  - 기대 결과: 오류 발생 항목은 로그에 기록하고 건너뛴 후, 다음 항목의 수집을 계속한다.
  - 예외 상황: 인증 실패, 네트워크 완전 단절 등 복구 불가능한 오류의 경우 스크립트를 종료한다.

## 6. 비기능 요구사항

### 6.1 안정성

- **NFR-001**: flex 서비스에 과도한 부하를 주지 않도록 요청 간 적절한 간격을 유지해야 한다.
- **NFR-002**: 네트워크 일시 오류 발생 시 재시도 로직을 포함해야 한다.

### 6.2 보안

- **NFR-003**: 수집된 데이터에는 개인정보(이름, 소속 등)가 포함될 수 있으므로, 저장된 파일에 대한 접근 관리는 실행자의 책임으로 한다.
- **NFR-004**: 인증 정보(비밀번호, 토큰 등)는 로그에 출력되지 않아야 한다.

### 6.3 호환성

- **NFR-005**: Node.js 20 이상 환경에서 실행 가능해야 한다.
- **NFR-006**: macOS 환경에서 실행 가능해야 한다.

## 7. 비즈니스 규칙

- **BR-001**: 수집 대상은 실행자의 flex 계정이 속한 단일 테넌트(회사)의 데이터로 한정한다.
- **BR-002**: 수집은 읽기 전용이어야 하며, flex 서비스의 데이터를 변경하는 어떠한 행위도 수행하지 않는다.
- **BR-003**: 실행자의 flex 권한으로 접근 가능한 범위의 데이터만 수집한다. 권한 밖의 데이터에 접근을 시도하지 않는다.
- **BR-004**: 동일한 데이터를 중복 수집하지 않는다. 전자결재 워크플로우와 근태/휴가 승인이 내부적으로 같은 데이터인 경우 한 번만 수집한다.

- 날짜/시간 값은 타임존 정보를 포함하거나, 어떤 타임존 기준인지 명시되어야 한다.
- 식별자는 같은 유형 내에서 고유해야 한다.
- 인스턴스의 양식 참조는 수집된 양식 데이터와 연결 가능해야 한다.

## 9. 화면/인터페이스 요구사항

본 스크립트는 CLI(명령줄 인터페이스)로 실행되며, 별도의 GUI를 제공하지 않는다.

### 9.1 입력

- 실행자는 스크립트 실행 전 인증 정보(계정 정보 또는 토큰)를 설정해야 한다.
- 스크립트 실행 시 별도의 인자나 옵션 입력 없이 실행 가능해야 한다.

### 9.2 출력

- 실행 중: 현재 진행 단계, 처리 건수 등의 진행 상황
- 실행 완료 후: 수집 결과 요약 (양식 수, 인스턴스 수, 실패 건수, 소요 시간, 저장 위치)
- 오류 발생 시: 오류 내용과 해당 항목 정보

## 10. 제약 사항 및 가정

### 10.1 제약 사항

- **C-001**: flex Open API의 워크플로우 데이터 제공 범위가 제한적일 수 있다. 현재 확인된 Open API 범위는 인사정보, 조직도, 근무/휴가 이력 중심이며, 전자결재 워크플로우 전용 API의 존재 여부는 미확인 상태이다.
- **C-002**: 웹 UI 크롤링에 의존하는 경우, flex 서비스의 UI 변경에 취약하다. 일회성 스크립트이므로 장기 유지보수는 고려하지 않는다.
- **C-003**: 실행자의 flex 계정 권한에 따라 수집 가능한 데이터 범위가 제한된다. 관리자 권한이 없는 경우 다른 구성원의 문서나 전체 양식 목록에 접근하지 못할 수 있다.
- **C-004**: 본 스크립트는 `apps/flex-crawler` 디렉토리 내에 위치한다.

### 10.2 가정

- **A-001**: 실행자는 flex 서비스에 로그인 가능한 유효한 계정을 보유하고 있다.
- **A-002**: 실행 환경에서 flex 서비스(flex.team)에 대한 네트워크 접근이 가능하다.
- **A-003**: 실행자의 flex 계정은 워크플로우 데이터를 조회할 수 있는 충분한 권한(관리자 이상)을 보유하고 있는 것으로 가정한다.
- **A-004**: flex 서비스의 이용 약관이 본 스크립트의 데이터 수집 행위를 금지하지 않는 것으로 가정한다.
