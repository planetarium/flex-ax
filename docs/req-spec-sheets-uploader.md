# flex-sheets-uploader 요구 명세서

## 1. 개요

### 1.1 배경 및 목적

flex-crawler는 flex HR SaaS의 워크플로우 데이터(양식 템플릿, 결재 문서 인스턴스, 근태/휴가 승인 이력)를 수집하여 JSON 파일로 저장한다. 그러나 JSON 파일은 비개발직군 구성원이 데이터를 열람하고 분석하기에 적합하지 않다.

본 요구사항은 flex-crawler가 생성한 JSON 데이터를 스프레드시트 형태로 변환하여 Google Sheets에 업로드하는 별도 앱(flex-sheets-uploader)을 정의한다. 이를 통해 수집된 워크플로우 데이터를 스프레드시트에서 쉽게 조회, 필터링, 분석할 수 있도록 한다.

### 1.2 이 요구사항이 해결하려는 문제

- JSON 파일은 비개발직군이 직접 열람하기 어렵다
- 개별 JSON 파일이 분산되어 있어 전체 현황을 한눈에 파악하기 어렵다
- 데이터에 대한 필터링, 정렬, 통계 분석 등의 작업이 JSON 상태에서는 불편하다

## 2. 범위

### 2.1 포함 범위 (In-Scope)

- flex-crawler output 디렉토리의 JSON 파일 읽기
- 세 가지 데이터 유형(템플릿, 인스턴스, 근태/휴가 승인)을 스프레드시트 형태로 변환
- 변환된 데이터를 Google Sheets에 업로드
- flex-ax 모노레포 내 `apps/flex-sheets-uploader` 경로에 별도 앱으로 구성

### 2.2 제외 범위 (Out-of-Scope)

- flex-crawler의 수집 로직 변경
- Google Sheets 이외의 스프레드시트 서비스(Excel Online, Notion 등) 지원
- 실시간 동기화 또는 자동 갱신 (수동 실행 방식만 포함)
- 업로드된 스프레드시트의 서식, 차트, 피벗 테이블 등 시각화 설정
- `_raw` 필드의 변환 (원시 데이터는 변환 대상에서 제외)

## 3. 용어 정의

| 용어 | 정의 |
|------|------|
| **템플릿 (Template)** | flex 워크플로우에서 결재 문서를 작성할 때 사용하는 양식 정의. 필드 목록, 카테고리, 기본 결재선 등을 포함한다. |
| **인스턴스 (Instance)** | 특정 템플릿을 기반으로 실제 작성된 결재 문서. 기안자, 작성일, 결재 상태, 필드 값, 첨부파일 정보, 결재선 등을 포함한다. |
| **근태/휴가 승인 (Attendance Approval)** | 근태 또는 휴가 관련 승인 이력. 신청자, 유형, 상태, 승인자 정보 등을 포함한다. |
| **크롤 리포트 (Crawl Report)** | flex-crawler 실행 결과 요약. 수집 시작/종료 시각, 각 유형별 수집 건수와 오류 정보를 포함한다. |
| **output 디렉토리** | flex-crawler가 수집 결과를 저장하는 디렉토리. 기본값은 flex-crawler 앱 기준 `./output`이며, 하위에 `templates/`, `instances/`, `attendance/`, `attachments/` 디렉토리와 `crawl-report.json` 파일이 존재한다. |
| **필드 (Field)** | 인스턴스에 포함된 사용자 입력 값. 필드명(fieldName), 필드 유형(fieldType), 값(value)으로 구성된다. |
| **결재선 (Approval Line)** | 결재 문서의 승인 절차를 구성하는 단계 목록. 각 단계는 순서, 유형, 승인자, 상태, 처리 일시 등을 포함한다. |

## 4. 사용자 및 역할

| 역할 | 설명 | 권한 범위 |
|------|------|-----------|
| **실행자** | flex-sheets-uploader를 CLI에서 실행하는 사용자 | 앱 실행, 설정(입력 경로, Google Sheets 인증 정보) 구성, 업로드 대상 스프레드시트 지정 |
| **스프레드시트 열람자** | 업로드된 Google Sheets를 조회하는 사용자 | Google Sheets의 공유 설정에 따라 열람 (본 앱의 범위 밖) |

## 5. 기능 요구사항

### 5.1 JSON 데이터 읽기

- **FR-001**: 시스템은 지정된 입력 경로(input path)에서 flex-crawler output 디렉토리 구조를 인식하고, 해당 디렉토리 내의 JSON 파일을 읽을 수 있어야 한다.
  - 선행 조건: 입력 경로에 flex-crawler가 생성한 output 디렉토리 구조가 존재해야 한다.
  - 기대 결과: `templates/`, `instances/` 하위의 모든 `*.json` 파일과 `crawl-report.json`을 성공적으로 읽는다.
  - 예외 상황:
    - 입력 경로가 존재하지 않는 경우: 명확한 오류 메시지와 함께 실행을 중단한다.
    - 디렉토리 내에 JSON 파일이 하나도 없는 경우: 해당 데이터 유형에 대해 빈 결과를 반환하고 경고 메시지를 출력한다.
    - JSON 파일의 형식이 올바르지 않은 경우(파싱 실패): 해당 파일을 건너뛰고, 파일명과 오류 원인을 경고로 기록한 후 나머지 파일은 계속 처리한다.

- **FR-002**: 시스템은 `crawl-report.json`을 읽어 수집 실행 정보(시작/종료 시각, 수집 건수, 오류 건수)를 확인할 수 있어야 한다.
  - 선행 조건: 입력 경로에 `crawl-report.json`이 존재한다.
  - 기대 결과: 리포트 정보를 파싱하여 업로드 시 메타데이터로 활용한다.
  - 예외 상황: `crawl-report.json`이 없는 경우, 경고 메시지를 출력하되 나머지 데이터 변환 및 업로드는 계속 진행한다.

### 5.2 스프레드시트 형태로 변환

- **FR-003**: 시스템은 템플릿 데이터를 다음 열(column)을 포함하는 행(row) 기반 형태로 변환해야 한다.
  - 필수 열: 템플릿 ID, 템플릿 이름, 카테고리, 필드 수, 생성일, 수정일
  - 각 템플릿은 하나의 행으로 표현된다.
  - `_raw` 필드는 변환 대상에서 제외한다.
  - 기대 결과: 29개의 템플릿 JSON이 29행의 표 형태 데이터로 변환된다.

- **FR-004**: 시스템은 템플릿별 필드 정의 데이터를 별도의 시트로 변환해야 한다.
  - 필수 열: 템플릿 ID, 템플릿 이름, 필드 이름, 필드 유형, 필수 여부, 옵션 목록, 설명
  - 하나의 템플릿이 여러 필드를 가진 경우, 필드별로 별도 행이 생성된다.
  - 기대 결과: 모든 템플릿의 필드 정의가 하나의 시트에 집약된다.

- **FR-005**: 시스템은 인스턴스(결재 문서) 데이터를 다음 열을 포함하는 행 기반 형태로 변환해야 한다.
  - 필수 열: 인스턴스 ID, 문서번호, 템플릿 ID, 템플릿 이름, 기안자 이름, 기안자 부서, 작성일, 결재 상태, 결재선 단계 수, 첨부파일 수
  - 각 인스턴스는 하나의 행으로 표현된다.
  - `_raw` 필드는 변환 대상에서 제외한다.

- **FR-006**: 시스템은 인스턴스의 필드 값 데이터를 별도의 시트로 변환해야 한다.
  - 필수 열: 인스턴스 ID, 문서번호, 필드 이름, 필드 유형, 필드 값
  - 하나의 인스턴스가 여러 필드 값을 가진 경우, 필드별로 별도 행이 생성된다.
  - 필드 값이 객체나 배열인 경우, 문자열로 직렬화하여 저장한다.

- **FR-007**: 시스템은 인스턴스의 결재선 데이터를 별도의 시트로 변환해야 한다.
  - 필수 열: 인스턴스 ID, 문서번호, 단계 순서, 승인 유형, 승인자 이름, 승인자 부서, 승인 상태, 처리 일시, 코멘트
  - 하나의 인스턴스가 여러 결재 단계를 가진 경우, 단계별로 별도 행이 생성된다.

- **FR-008**: 시스템은 근태/휴가 승인 데이터를 다음 열을 포함하는 행 기반 형태로 변환해야 한다.
  - 필수 열: 승인 ID, 유형, 신청자 이름, 신청자 부서, 신청일, 상태, 승인자 이름, 처리 일시
  - `details` 필드의 각 키-값 쌍은 문자열로 직렬화하여 단일 열에 포함한다.
  - `_raw` 필드는 변환 대상에서 제외한다.
  - 예외 상황: 근태/휴가 승인 데이터가 없는 경우(수집 건수 0), 해당 시트는 헤더 행만 포함하여 생성하거나 생성을 생략한다.

- **FR-009**: 시스템은 크롤 리포트를 별도의 시트로 변환해야 한다.
  - 필수 항목: 수집 시작 시각, 수집 종료 시각, 총 소요 시간, 각 유형별(템플릿/인스턴스/근태) 전체 건수, 성공 건수, 실패 건수, 총 오류 건수
  - 오류가 있는 경우, 오류 목록(대상, 단계, 메시지, 시각)도 포함한다.

### 5.3 Google Sheets 업로드

- **FR-010**: 시스템은 변환된 모든 시트 데이터를 하나의 Google Sheets 스프레드시트에 업로드할 수 있어야 한다.
  - 선행 조건: 유효한 Google Sheets 인증 정보가 구성되어 있어야 한다.
  - 기대 결과: 하나의 Google Sheets 문서 내에 데이터 유형별 시트(탭)가 생성되고, 각 시트에 헤더 행과 데이터 행이 기록된다.
  - 예외 상황:
    - 인증 실패: 오류 원인을 명시하는 메시지를 출력하고 실행을 중단한다.
    - 네트워크 오류: 오류 메시지를 출력하고 실행을 중단한다.

- **FR-011**: 시스템은 새 Google Sheets 스프레드시트를 생성하여 업로드하는 모드를 지원해야 한다.
  - 기대 결과: 새로운 스프레드시트가 생성되며, 생성된 스프레드시트의 URL이 사용자에게 출력된다.
  - 스프레드시트 이름은 기본적으로 수집 일시를 포함하여 자동 생성되며, 사용자가 별도로 지정할 수도 있어야 한다.

- **FR-012**: 시스템은 기존 Google Sheets 스프레드시트를 지정하여 데이터를 덮어쓰는 모드를 지원해야 한다.
  - 선행 조건: 대상 스프레드시트 ID 또는 URL이 설정으로 제공된다.
  - 기대 결과: 기존 스프레드시트의 해당 시트 내용이 새 데이터로 교체된다.
  - 예외 상황: 지정된 스프레드시트가 존재하지 않거나 접근 권한이 없는 경우, 명확한 오류 메시지를 출력하고 실행을 중단한다.

- **FR-013**: 각 시트의 첫 번째 행은 열 이름(헤더)으로 구성되어야 한다.
  - 기대 결과: 모든 시트에서 첫 행은 데이터가 아닌 열 제목이며, 두 번째 행부터 실제 데이터가 시작된다.

### 5.4 실행 및 설정

- **FR-014**: 시스템은 CLI 명령어로 실행할 수 있어야 한다.
  - 기대 결과: 모노레포의 scripts 컨벤션에 따라 `start` 스크립트로 실행 가능하다.

- **FR-015**: 시스템은 다음 설정 항목을 지원해야 한다.

  | 설정 항목 | 필수 여부 | 설명 |
  |-----------|-----------|------|
  | 입력 경로 | 필수 | flex-crawler output 디렉토리 경로 |
  | Google Sheets 인증 정보 | 필수 | Google Sheets에 접근하기 위한 인증 정보 |
  | 대상 스프레드시트 ID | 선택 | 기존 스프레드시트에 업로드할 경우 해당 스프레드시트 식별자. 미지정 시 새 스프레드시트 생성 |
  | 스프레드시트 이름 | 선택 | 새로 생성할 스프레드시트의 이름. 미지정 시 자동 생성 |

  - 예외 상황: 필수 설정이 누락된 경우, 누락 항목을 명시하는 오류 메시지를 출력하고 실행을 중단한다.

- **FR-016**: 시스템은 실행 진행 상황을 사용자에게 출력해야 한다.
  - 기대 결과: 최소한 다음 단계에서 로그 메시지가 출력된다.
    - 설정 로딩 완료
    - 입력 데이터 읽기 시작/완료 (읽은 파일 수)
    - 데이터 변환 시작/완료 (변환된 행 수)
    - Google Sheets 업로드 시작/완료
    - 최종 결과 요약 (스프레드시트 URL, 처리 건수)

- **FR-017**: 시스템은 실행 완료 후 업로드된 Google Sheets의 URL을 출력해야 한다.
  - 기대 결과: 사용자가 출력된 URL을 통해 즉시 스프레드시트에 접근할 수 있다.

### 5.5 모노레포 통합

- **FR-018**: 시스템은 flex-ax 모노레포의 `apps/` 디렉토리 하위에 `flex-sheets-uploader`라는 이름의 독립 패키지로 존재해야 한다.
  - 기대 결과: `apps/flex-sheets-uploader/package.json`이 존재하며, 모노레포의 워크스페이스로 인식된다.

- **FR-019**: 시스템은 기존 모노레포 컨벤션(TypeScript, ESM, pnpm, turbo)을 따라야 한다.
  - 기대 결과: `build`, `lint`, `clean` 등 기존 모노레포 스크립트 체계와 호환된다.

## 6. 비기능 요구사항

### 6.1 성능

- **NFR-001**: 현재 규모(템플릿 29건, 인스턴스 7건)의 데이터에 대해, 전체 처리(읽기 + 변환 + 업로드)가 60초 이내에 완료되어야 한다.
- **NFR-002**: 향후 데이터 규모가 증가할 경우(템플릿 수백 건, 인스턴스 수천 건)에도 합리적인 시간 내에 처리가 완료되어야 한다. Google Sheets API의 요청 제한을 초과하지 않도록 한다.

### 6.2 보안

- **NFR-003**: Google Sheets 인증 정보(서비스 계정 키 등)는 소스 코드에 포함되지 않아야 하며, 환경 변수 또는 외부 설정 파일을 통해 제공되어야 한다.
- **NFR-004**: 인증 정보 파일의 경로가 `.gitignore`에 포함되어야 한다.

### 6.3 신뢰성

- **NFR-005**: 개별 JSON 파일의 파싱 실패가 전체 프로세스를 중단시키지 않아야 한다. 실패한 파일은 건너뛰고 나머지를 계속 처리하며, 실패 목록을 최종 결과에 포함해야 한다.

## 7. 비즈니스 규칙

- **BR-001**: 하나의 Google Sheets 스프레드시트 내에 데이터 유형별로 별도의 시트(탭)를 생성한다. 최소한 다음 시트가 생성되어야 한다:
  1. 크롤 리포트 요약
  2. 템플릿 목록
  3. 템플릿 필드 정의
  4. 인스턴스 목록
  5. 인스턴스 필드 값
  6. 인스턴스 결재선
  7. 근태/휴가 승인 목록 (데이터가 존재하는 경우)

- **BR-002**: 날짜/시각 데이터는 ISO 8601 형식으로 표시한다.

- **BR-003**: `_raw` 필드는 모든 데이터 유형에서 변환 대상에서 제외한다. `_raw`는 flex API의 원시 응답으로 디버깅 목적으로만 보관된다.

- **BR-004**: 인스턴스의 필드 값 중 객체 또는 배열 형태의 값은 사람이 읽을 수 있는 문자열로 직렬화한다.

- **BR-005**: 스프레드시트 생성 시 이름을 지정하지 않은 경우, `flex-workflow-data-YYYY-MM-DD` 형식으로 자동 생성한다. (예: `flex-workflow-data-2026-03-31`)

## 8. 데이터 요구사항

### 8.1 입력 데이터

시스템이 읽어야 하는 입력 데이터의 구조는 다음과 같다:

#### 8.1.1 디렉토리 구조

```
{input_path}/
  crawl-report.json
  templates/
    {templateId}.json    (0개 이상)
  instances/
    {instanceId}.json    (0개 이상)
  attendance/
    {approvalId}.json    (0개 이상)
  attachments/           (본 앱에서는 읽지 않음)
```

#### 8.1.2 템플릿 JSON 데이터 항목

| 항목 | 유형 | 설명 |
|------|------|------|
| id | 문자열 | 템플릿 고유 식별자 |
| name | 문자열 | 템플릿 이름 |
| category | 문자열 (선택) | 카테고리 |
| fields | 배열 | 필드 정의 목록. 각 필드는 name(문자열), type(문자열), required(불리언, 선택), options(문자열 배열, 선택), description(문자열, 선택)을 포함 |
| defaultApprovalLine | 배열 (선택) | 기본 결재선 |
| createdAt | 문자열 (선택) | 생성 일시 |
| updatedAt | 문자열 (선택) | 수정 일시 |
| permissions | 객체 (선택) | 권한 정보 |
| _raw | 임의 (선택) | 원시 데이터 - 변환 제외 대상 |

#### 8.1.3 인스턴스 JSON 데이터 항목

| 항목 | 유형 | 설명 |
|------|------|------|
| id | 문자열 | 인스턴스 고유 식별자 |
| documentNumber | 문자열 (선택) | 문서 번호 (예: "2024-1791") |
| templateId | 문자열 | 사용된 템플릿 ID |
| templateName | 문자열 | 사용된 템플릿 이름 |
| drafter | 객체 | 기안자 정보. name(문자열, 필수), id/department/position(문자열, 선택) |
| draftedAt | 문자열 | 작성 일시 |
| status | 문자열 | 결재 상태 (예: DONE, DECLINED, pending, in_progress, approved, rejected, canceled) |
| approvalLine | 배열 | 결재선 단계 목록. 각 단계는 order, type, approver(UserInfo), status, processedAt(선택), comment(선택) |
| fields | 배열 | 필드 값 목록. 각 필드는 fieldName, fieldType, value를 포함 |
| attachments | 배열 | 첨부파일 정보 목록. 각 항목은 fileName, fileSize(선택), mimeType(선택), localPath(선택) |
| modificationHistory | 배열 (선택) | 수정 이력 목록 |
| _raw | 임의 (선택) | 원시 데이터 - 변환 제외 대상 |

#### 8.1.4 근태/휴가 승인 JSON 데이터 항목

| 항목 | 유형 | 설명 |
|------|------|------|
| id | 문자열 | 승인 고유 식별자 |
| type | 문자열 | 승인 유형 |
| applicant | 객체 | 신청자 정보. name(문자열, 필수), id/department/position(문자열, 선택) |
| appliedAt | 문자열 | 신청 일시 |
| details | 객체 | 세부 정보 (키-값 쌍, 구조 가변적) |
| status | 문자열 | 승인 상태 |
| approver | 객체 (선택) | 승인자 정보 |
| processedAt | 문자열 (선택) | 처리 일시 |
| _raw | 임의 (선택) | 원시 데이터 - 변환 제외 대상 |

### 8.2 데이터 유효성 규칙

- 템플릿 JSON은 최소한 `id`와 `name`이 존재해야 유효한 것으로 간주한다.
- 인스턴스 JSON은 최소한 `id`, `templateId`, `templateName`, `drafter`, `draftedAt`, `status`가 존재해야 유효한 것으로 간주한다.
- 근태/휴가 승인 JSON은 최소한 `id`, `type`, `applicant`, `appliedAt`, `status`가 존재해야 유효한 것으로 간주한다.
- 유효성 검사에 실패한 파일은 FR-001의 예외 상황 규칙에 따라 건너뛴다.

## 10. 제약 사항 및 가정

### 10.1 제약 사항

- Google Sheets API의 셀 수 제한(스프레드시트당 최대 1,000만 셀)을 초과하지 않아야 한다.
- Google Sheets API의 요청 빈도 제한(분당 요청 수)을 준수해야 한다.
- 본 앱은 flex-crawler에 대한 런타임 의존성을 갖지 않는다. flex-crawler의 출력 결과물(JSON 파일)만을 입력으로 사용한다.

### 10.2 가정

- flex-crawler가 정상적으로 실행되어 output 디렉토리에 JSON 파일이 생성된 상태에서 본 앱이 실행된다.
- 실행자는 Google Cloud 프로젝트에서 Google Sheets API가 활성화된 서비스 계정 또는 OAuth 인증 정보를 사전에 준비한다.
- 실행 환경은 Node.js 20 이상이다 (모노레포의 `engines` 설정에 따름).
- 한 번의 실행으로 전체 데이터를 업로드하며, 증분(incremental) 업로드는 지원하지 않는다.
